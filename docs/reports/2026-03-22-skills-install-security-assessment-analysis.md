# 2026-03-22 skills 安装安全评估（高风险 / 致命风险）分析报告

## 背景与数据来源

本报告基于在本地执行 `skills add` 安装 `ai-plugins/common-tools/skills` 时，安装器输出的 **Security Risk Assessments** 摘要，并结合仓库内技能源码（尤其 `use-other-model`）做的说明性分析，供后续是否调整技能内容、安装策略或文档时参考。

**分析目的**：解释为何部分技能被标为 **High Risk**、**Critical Risk**，以及这些标签在自动化扫描语境下的真实含义，**不等同于**「技能包已被判定为恶意软件」。

---

## 1. 终端输出中的评估维度

安装器在「Installation Summary」之后展示的安全评估表，常见列含义如下（行业常见理解，具体以 skills.sh / 安装器文档为准）。

| 列名       | 含义概述                                                                       |
| ---------- | ------------------------------------------------------------------------------ |
| **Gen**    | 对技能文本或行为模式的启发式 / 模型辅助判断，例如 Safe、High Risk。            |
| **Socket** | 与依赖、供应链或敏感 API 使用模式相关的告警统计，**alerts** 多为规则命中条数。 |
| **Snyk**   | Snyk 等工具给出的严重度分级，例如 Low、Med、**Critical**。                     |

上述结果通常为对技能包内容（Markdown、示例脚本、引用路径等）的**静态与规则扫描**，不是对「用户机器已被入侵」的实时检测。

---

## 2. 案例回顾：`use-other-model` 为何呈现最高档风险

在典型安装输出中，`use-other-model` 可能出现类似组合：

- Gen：**High Risk**
- Socket：**3 alerts**
- Snyk：**Critical Risk**

而同批安装的其它技能多为 Safe / Low，个别为 Med 或单项 Socket 告警。

### 2.1 技能本身在教用户做什么

`use-other-model` 的设计目标是：指导主代理将部分任务委托给其它 AI 模型（如 MiniMax、Gemini），通过 MCP 或独立 Claude Code 会话降低 token 成本。因此文档中**必然**包含：

- 外部 API / MCP 调用说明；
- 子进程、Bash、后台任务（`run_in_background`）等流程；
- 与凭证、环境变量相关的配置说明。

这类「编排高权限代理行为」的技能，在静态扫描中几乎总是比「纯文档 / 纯配置初始化」类技能得分更高。

### 2.2 易被安全规则命中的具体模式（仓库内确有对应表述）

以下模式在 `use-other-model` 的 `references/code-templates.md`、`references/method-b-independent-session.md` 等文件中出现，**会显著推高** Gen / Snyk / Socket 的评分：

1. **`unset CLAUDECODE`**  
   文档中用于说明如何绕过 Claude Code 对嵌套会话的限制。对安全产品而言，属于「规避产品内置防护」的典型文本模式。

2. **`claude --dangerously-skip-permissions`**  
   标志名与语义均表示**跳过权限确认**。静态规则几乎必然将其标为严重级别。

3. **Shell 中导出 API 凭证类环境变量**  
   例如 `ANTHROPIC_AUTH_TOKEN`、Base URL 等与第三方端点组合，易触发密钥泄露 / 供应链类规则（即便仅为模板占位）。

4. **独立进程、后台任务、Heredoc 传参**  
   描述「拉起长时间运行、高自主度的子会话」，在行为分类上属于高影响面操作。

### 2.3 结论（针对「致命风险」字样）

**Critical / High Risk 在此语境下主要表示：若严格按文档中的示例执行，操作具备「高权限、可绕过确认、可处理密钥、可启动子进程」等特征。**

自动化扫描通常**不会**区分「恶意植入」与「带风险说明的合法教学文档」，只要文本命中上述模式，就可能给出最高档标签。

## 3. 同批其它技能的差异（简要）

| 现象                                                       | 可能原因（概括）                                                                      |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 多数技能为 Safe + Low                                      | 内容以 Git 工作流、编辑器配置、提交规范等为主，少见「跳过权限」「嵌套绕过」等强信号。 |
| `git-commit` 等为 Med Risk                                 | 与命令执行、提交行为、钩子等相关时，常比纯文档略高。                                  |
| `init-claude-code-statusline` 等出现 **1** 条 Socket alert | 单条模式命中（如某类依赖引用、网络或配置示例），整体仍可能标为 Gen Safe。             |

**注意**：若安装器报告安装 **9** 个技能，而安全评估表仅列出 **8** 个，可能存在展示省略或「无告警项不展示」等情况；完整明细应以安装器给出的 **Details** 链接或官方报告为准。

## 4. 如何理性使用本报告结论

1. **标签含义**：反映的是「技能所描述的操作模型」在自动化规则下的敏感程度，**不是**单独的「漏洞 CVE 编号」或「恶意软件判定」。
2. **实际风险**：取决于用户是否在受控环境执行、是否妥善管理 API Key、是否滥用 `--dangerously-skip-permissions` 等；技能内若已有「勿将密钥写入 prompt、执行后删除含密钥脚本」等说明，与扫描器「高危」标签在语义上可并存。
3. **后续可做事项**（仅列方向，本报告不实施改动）：若希望降低静态扫描分数，通常需调整示例表述、占位符策略、敏感标志的展示方式等，并与产品安全要求平衡。

## 6. 报告小结

`use-other-model` 被标为高风险与 Snyk 致命级，**主要原因是文档与模板中集中出现了绕过嵌套检测、跳过权限、Shell 导出 Token、子进程与后台任务等模式**；这些是安全扫描对「高影响代理行为」的典型命中，**不等于**该技能包本身含有恶意代码。本报告供你结合业务需求与安全策略做进一步分析或决策。

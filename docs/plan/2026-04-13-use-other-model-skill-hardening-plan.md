# 2026-04-13 use-other-model 技能加硬升级计划

## 目标

升级 `use-other-model` 技能，让主代理在驱动本地 `claude code` 外部会话时更稳定、更少误判，重点解决以下问题：

1. 一次性授予可编辑权限，避免反复试错启动参数。
2. 明确传递完整任务上下文，避免子会话把任务误识别为普通问答。
3. 对长任务预置足够的执行时长与预算，不再因为默认时长过短中断。
4. 把浏览器验收纳入强制环节，避免只跑 `build/test` 就误判为“完成”。

---

## 当前缺陷

### 1. 启动参数不够硬

当前技能虽然覆盖了独立 Claude Code 会话的基本思路，但没有把以下参数收敛成默认模板：

- `--permission-mode bypassPermissions`
- `--tools default`
- `--append-system-prompt` 的无人值守执行约束
- 合适的 `timeout_ms`

结果是主代理需要临场试错，才能让外部会话真正进入可编辑执行态。

### 2. Prompt 仍然偏“任务说明”，不够“代理指令”

目前的技能更像在教主代理“把事情交给另一个模型”，但没有明确要求：

- 外部会话不得反问
- 必须先读指定文件
- 必须自己执行命令
- 必须自己验证
- 完成后必须退出

这会让子会话把请求当成普通聊天，而不是 unattended coding session。

### 3. 上下文打包不完整

当前技能没有强制要求主代理提供一份“上下文封包”：

- 仓库根目录
- 当前分支
- 必读文件清单
- 允许修改的文件范围
- 禁止操作清单
- 验收标准
- 运行命令

缺少这份封包时，子会话容易漏掉真正关键的 guardrail。

### 4. 缺少浏览器验收闭环

当前技能强调了日志、测试、构建，但没有把以下能力变成默认要求：

- 启动或复用本地 dev server
- 打开 `http://localhost:5173/#/`
- 进行至少一次视觉检查
- 进行至少一次交互检查
- 把浏览器观察写回执行日志

这正是这次大屏重构里最明显的缺口。

### 5. 缺少失败分流策略

当前技能没有把失败分成三类处理：

1. 启动失败
2. 编辑成功但验证失败
3. 浏览器视觉验收失败

没有分流时，主代理容易过早汇报“完成”，或者在错误层面上做无效修补。

---

## 升级原则

### 原则 1：默认按“无人值守编码代理”设计

技能文档应明确：

- 方案 B 不是“再开一个问答会话”
- 而是“启动一个具备编辑、执行、验证责任的独立 coding agent”

所有模板都要围绕这个定位来写。

### 原则 2：默认给足约束，而不是临场补约束

必须把常用的执行权限、工具、系统提示、日志、预算、超时写进模板。
不再让主代理在执行中补开关。

### 原则 3：浏览器验收是复杂前端任务的默认步骤

只要任务涉及页面、组件、样式、交互、可视化，就要把浏览器访问纳入执行模板。

### 原则 4：主代理永远保留复核责任

即便子会话完成编码，主代理仍必须：

- 重新看 `git diff`
- 重新跑关键命令
- 重新看浏览器效果

技能文档应把这条写成硬约束，而不是建议。

## 计划中的升级项

## Task 1：新增标准启动参数模板

为技能补充一份“PowerShell / Bash 双模板”，默认包含：

- `--permission-mode bypassPermissions`
- `--tools default`
- `--output-format json`
- `--append-system-prompt "<硬约束提示>"`
- 大于 15 分钟的命令超时建议

要求：

- 不再把权限模式作为可选建议
- 默认就是可编辑、可执行、可验证模式
- 仅在高风险场景下再显式降权

## Task 2：新增“上下文封包”模板

在 `use-other-model` 的方案 B 中新增固定章节：

1. Working directory
2. Branch
3. Read first
4. Allowed edits
5. Do not do
6. Verification commands
7. Browser verification target
8. Required output log
9. Completion rule

要求主代理在启动外部会话前，必须先写好这份任务封包文件。

## Task 3：新增“无人值守系统提示”模板

补充标准的 `append system prompt` 模板，至少覆盖：

- 你是 unattended coding agent
- 不要反问，除非任务封包本身冲突
- 先读文件，再改代码，再跑命令，再做验证
- 对前端任务必须做浏览器检查
- 完成后立即退出

这部分应该从技能文档中的“建议提示”升级为“标准提示模板”。

## Task 4：把浏览器验收写入默认工作流

为前端/UI 类任务增加强制步骤：

1. 打开本地 URL
2. 观察首屏是否加载
3. 检查是否有明显布局错误
4. 执行至少一个核心交互
5. 将观察结果写入 execution log

如果浏览器不可用，也必须在日志里记录“为何未验证”，不能静默跳过。

## Task 5：新增执行时长与预算指引

在技能里增加一段明确规则：

- 简单任务：5 分钟以内
- 中等任务：10 至 20 分钟
- 复杂任务：20 至 45 分钟

并为方案 B 的主代理提供默认超时表：

- 启动会话：2 分钟
- 编码执行：30 分钟
- 前端构建/测试：5 分钟
- 浏览器验收：5 分钟

避免主代理沿用过短超时导致误判“模型不工作”。

## Task 6：新增失败分流与回退策略

文档需要明确：

### 启动失败

- 优先检查 `claude --help`
- 检查是否支持对应参数
- 检查 provider 环境变量
- 检查权限模式和工具模式

### 执行失败

- 查看 execution log
- 查看 stdout/stderr
- 判断是编译错误、测试错误、还是运行错误

### 浏览器验收失败

- 记录具体视觉问题
- 回到子会话继续迭代，或者由主代理直接补刀

### 连续两轮失败

- 停止继续使用外部模型
- 主代理接管实施

## Task 7：补充“前端任务专用模板”

为前端任务单独增加一个模板，不再复用通用文本。
模板里必须包含：

- 指定页面 URL
- 指定浏览器验证
- 指定视觉对比目标
- 指定布局/交互验收点

这样像大屏、后台、表单、地图页这类任务，就不会只靠 `build/test` 判断完成。

## 验收标准

技能升级完成后，应满足以下结果：

1. 主代理首次生成的外部会话启动命令即可直接进入可编辑模式。
2. 外部会话不会再把复杂任务误识别为普通问答。
3. 前端任务默认包含浏览器访问与交互验收。
4. 主代理知道何时回退，不会在错误执行模式上持续打补丁。
5. 技能文档本身能直接复制模板使用，而不是仍需临场发明参数组合。

这样升级后，`use-other-model` 才能真正稳定地驱动本地 `claude code`，而不是靠主代理临场试错把命令“加硬”。

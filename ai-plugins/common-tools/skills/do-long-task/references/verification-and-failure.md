# 验证、失败与完成标记

本文件用于验证纪律、失败记录、连续失败阻塞和完成勾选。

## 验证纪律

任务必须验证后才能完成。

验证分为四层：

1. task-level verification
2. change-level verification
3. `/opsx:verify <change-name>` 或等价 OpenSpec strict validate
4. regression check

如果验证命令不存在，必须记录为“不可用”，不得伪造通过。

如果无法自动验证，必须写明替代验证方式和剩余风险。

## 完成标记纪律

只有满足以下条件，才能把 task 标记为 `[x]`：

- 代码已实现。
- task 验收标准满足。
- 相关验证命令通过，或替代验证已明确记录。
- OpenSpec strict validate 通过，或说明该项目没有对应命令。
- 验证结果写入 `agent-progress.md`。
- 重要风险写入 `agent-findings.md`。
- 没有未解决的 CRITICAL 问题。

不得因为“代码已经写完”就标记完成。

## 失败记忆纪律

失败必须成为长期记忆。

以下情况必须写入 `agent-findings.md`：

- 测试失败。
- 构建失败。
- 类型检查失败。
- lint 失败。
- validate 失败。
- 连续修复失败。
- 方案被证明不可行。
- 需求或设计冲突。
- 需要用户决策。
- 存在破坏性风险。

## 连续失败规则

- 第 1 次失败：分析并修复。
- 第 2 次同类失败：记录失败并换方案。
- 第 3 次同类失败：停止当前 task，输出 BLOCKED 报告。

不得无记录地重复尝试同一路径。

## BLOCKED 报告

触发阻塞时，输出：

- 当前 task。
- 已尝试路径。
- 失败证据。
- 已记录到哪个文件。
- 为什么不能继续。
- 需要用户提供的具体决策或外部条件。

不要把“工作很难”“验证较慢”“还没完全理解”当作阻塞；只有真正无法继续推进时才 BLOCKED。

## 完成前检查

汇报完成前确认：

- 代码已经实现。
- 验收标准已经满足。
- 相关验证命令已经通过，或者已记录替代验证。
- `agent-progress.md` 记录本轮进展和验证结果。
- `agent-findings.md` 记录重要发现、失败尝试或剩余风险。
- 没有未解决的 CRITICAL 问题。

# 常见问题

## Q1: 如何判断任务是否适合委托?

按三个问题快速判断:

1. 任务是否超过 5 分钟?
2. 能否写出清晰的任务封包?
3. 能否定义可执行的验收标准?

三个问题都回答“是”,才适合继续考虑委托。

## Q2: 方案 A 和方案 B 如何选择?

| 任务特征           | 推荐方案 |
| ------------------ | -------- |
| 单次调用、简单任务 | 方案 A   |
| 多步骤、复杂任务   | 方案 B   |
| 执行时间 < 2 分钟  | 方案 A   |
| 执行时间 > 5 分钟  | 方案 B   |
| 需要实时用户交互   | 方案 A   |
| 需要明确执行封包   | 方案 B   |

## Q3: 为什么方案 B 现在强调“无人值守编码代理”?

因为旧写法容易把外部会话启动成普通问答模式。

一旦子会话开始反问、泛泛总结、只给建议不执行,方案 B 的 token 优势和执行价值就会大幅下降。

## Q4: 为什么默认改用 `--permission-mode bypassPermissions`?

因为方案 B 的目标就是让子会话一次性进入可编辑、可执行、可验证状态。

如果仍把权限放在“临场试错”层面,主代理每次都要花时间调参,反而破坏了这项技能的价值。

## Q5: 什么时候还可以退回 `--dangerously-skip-permissions`?

只有在以下条件同时满足时:

1. 先用 `claude --help` 确认当前环境不支持 `--permission-mode`
2. 任务环境本身是受控沙箱
3. 主代理在执行日志中记录了为什么退回兼容模式

它应该是兼容回退,不是新的默认值。

## Q6: 如果用户没有配置 API 密钥怎么办?

1. 向用户索要必要的 provider 配置
2. 提供配置帮助
3. 如果用户不想配置,回退到直接执行
4. 不要强推其他模型

## Q7: 前端任务为什么不能只看 `build/test`?

因为页面类任务最常见的问题是:

- 布局错位
- 组件未渲染
- 交互无响应
- 视觉效果和预期不符

这些问题很多不会被 `build/test` 捕获。

## Q8: 子会话的 execution log 至少应该写什么?

至少写:

1. 开始时间
2. 已读文件
3. 实际修改
4. 运行过的命令
5. 浏览器观察
6. 遇到的问题
7. 完成时间
8. 完成结论或失败结论

任务封包模板见 `context-packet-template.md`。

## Q9: 浏览器不可用时怎么办?

不能静默跳过。

必须在 execution log 里写清楚:

1. 为什么浏览器不可用
2. 哪些浏览器验收项未执行
3. 当前只能给出什么级别的结论

这类结果应视为“验证不完整”,而不是“已完成”。

## Q10: 子会话执行失败后如何处理?

按 `failure-routing.md` 分流:

1. 启动失败 → 检查 `claude --help`、参数、环境变量
2. 执行失败 → 看 JSON 结果、execution log、stdout/stderr
3. 浏览器失败 → 记录具体问题,决定继续委托还是本地接管
4. 连续两轮失败 → 停止委托,主代理直接接管
